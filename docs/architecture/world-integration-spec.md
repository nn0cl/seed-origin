## システムの位置づけ

この機能は、権威型ゲームサーバーと局所環境シミュレーションのハイブリッドです。プレイヤーのログイン、魔法、攻撃、対象への効果適用、状態配信をサーバーが決定しながら、3次元の環境エーテルは減衰・伝播し、プレイヤーや対象物の行動で局所的に変化します。

クライアントは行動要求を送り、サーバーのシミュレーションコアが時刻、距離、リソース、環境状態を使って結果を決定します。魔法の結果は行動入力だけでは決まらず、発動時点の周辺エーテル、属性間相互作用、対象物の状態を含むワールド状態に依存します。

ここでいう決定論的処理とは、「同じ時点のワールド状態、同じ入力イベント列、同じ設定を与えれば、同じ結果になる」という意味です。火属性の使用が蓄積して周辺の火エーテル濃度や伝導率が変化すれば、後続の同じ火属性魔法の威力や影響範囲が変わります。これはランダムな結果ではなく、過去のイベントによってワールド状態が変化した結果です。

属性間の相互作用によって環境バランスが崩れた場合は、別属性の伝播、対象物の状態異常、地形や設備への悪影響などの二次イベントを発生させます。二次イベントも発生時刻とイベント連番を持ち、同時刻の処理順を固定します。

ニューラルネットワークは初期実装の前提にしません。将来導入する場合も、NPCの行動選択、パラメータ推定、異常検知などの補助用途に限定し、プレイヤー・対象物・エーテルの正解状態はシミュレーションコアが保持します。

推奨する論理境界は次のとおりです。

1. `CommandIngress`: ログイン、移動、魔法、攻撃などの要求を受け付ける。
2. `WorldSimulation`: 固定タイムステップで要求を検証し、イベントを適用する。
3. `EntityState`: プレイヤー、対象物、NPCの状態を保持する。
4. `EtherField`: 3次元の属性エーテル、減衰、伝播、伝導率を管理する。
5. `EffectSystem`: 命中結果、ダメージ、状態異常、環境注入を処理する。
6. `Snapshot/Events`: クライアントへ状態差分やイベントを配信する。

ネットワーク層は `WorldSimulation` の外側に置き、テストではネットワークなしで同じ入力列を実行できるようにします。

## 仕様の要約

### ワールドライフサイクル

`World` はシングルトンとして取得され、初期化時に次を行います。

1. `WorldStatus` を生成する。
2. 座標 `-10` 以上 `10` 未満の各軸を持つ、20 × 20 × 20 のグリッドを生成する。
3. 10ミリ秒間隔のワーカースレッドを開始する。
4. 各周期でグリッド環境を循環させ、最大60件のアクションを処理する。

設定ファイル `scode.ini` には `time_span=10` があるものの、現行 `World` は値を読まず、10ミリ秒を直接使用しています。取り込み時は設定値を正式な周期として扱う方針です。

### グリッド環境

各 `Grid` は次を保持します。なお、以下の`WorldGrid`案は将来の空間伝播仕様であり、現在のv1実装の属性名・相互作用とは区別します。

- 3次元座標
- 基準元素量
- `fire` / `water` / `earth` / `air` の元素量

v1の`EnvironmentEther`は全体値と`fire↔water`、`earth↔air`の対属性反作用を採用する。循環伝播と6方向隣接セルへの拡散は、`WorldGrid`導入時の候補であり、v1では実装しない。

各グリッドの元素値は時間経過に応じて減衰します。属性ごとに減衰率を設定可能にし、最低値を下回らないようにします。魔法や攻撃によるエーテル注入は、減衰との適用順序を固定して決定的に処理します。

#### 属性伝導率

環境エーテルの濃度が基準状態から高い方向または低い方向へ変化すると、その属性の伝導率が高まる仕様とします。初期モデルは次の式です。

`conductivity[a] = clamp(baseConductivity[a] + gain[a] × abs(ether[a] - baseline[a]), min, max)`

`a` は fire / water / earth / air のいずれかです。属性エーテルが濃すぎる場合だけでなく、枯渇している場合も、その属性の作用を伝えやすくします。v1の具体式は[`ether-rules.md`](ether-rules.md)に定義します。

伝導率は、魔法の環境拡散量、対象物への作用量、または隣接セルへの伝播量のいずれか一箇所へ適用します。同じ効果へ二重適用しないことを受入条件とします。

#### 魔法・攻撃による環境作用

魔法または攻撃は次のイベントとして扱います。

1. 実行者のリソース、距離、コマンド条件を検証する。
2. 対象プレイヤーまたは対象物へ直接効果を適用する。
3. 使用した魔法のエーテルを発生地点または対象地点へ注入する。
4. 発生地点から周囲へ、対象属性の伝導率に応じて拡散する。

対象物はプレイヤーとは別のエンティティとして扱えるようにし、地形・設備・資源にも同じ効果イベントを適用できる設計にします。

`scode` の `Effect.java` には `cRate`、`thval`、`scale` の試作がありますが、`World` の周期処理には接続されていません。また、`reduce()` が減衰後の元素値を新しい `Status` へ設定していないため、現状を完成仕様とはみなしません。

総元素量が基準値を超える場合は、6方向の隣接グリッドへ元素を拡散します。既存C++の `Field` はプレイヤーとアクションキューを担当しているため、グリッド環境は `WorldGrid` 相当の独立コンポーネントとして追加するのが安全です。

### プレイヤーと効果

プレイヤーはIDをキーにワールドへ登録されます。ログイン時は追加、ログアウト時は削除します。アクションが成立すると、対象プレイヤーへ `ActiveEffect` を追加します。

効果は次の情報を持ちます。

- 残り寿命
- 付与する `Status`

周期処理では効果を適用し、寿命を減らし、寿命切れの効果を削除します。これは既存C++の `Buff` と近い責務ですが、寿命と周期処理を明示的に持たせる必要があります。

### アクション実行条件

通常アクションは `ActionType.ACTION` の場合にキューへ入り、管理アクションは即時処理されます。

通常アクションの実行条件は次のとおりです。

1. コマンドIDからコマンド定義を取得できる。
2. 実行者のHP・MP・元素量がコマンド制限以上である。
3. 実行者と対象者の距離がコマンドの距離制限以内である。
4. 条件を満たした場合、対象者へコマンドの効果を付与する。

距離は3次元ユークリッド距離とします。

`Command` は少なくとも次を持ちます。

- ID、名称
- 距離制限
- 実行制限 `Status`
- 付与値 `Status`
- 効果寿命
- 詠唱時間（現行実装では未使用）

### ワールド状態

`WorldStatus` は次の監視値をJSON化できます。

- `ucount`: 現在のプレイヤー数
- `average`: 平均値用の未実装フィールド
- `uptime`: プロセス起動からの経過時間
- `instances`: インスタンス数用の未実装フィールド

`average` と `instances` は仕様未確定のため、C++移植の初期段階では予約フィールドとして保持します。

## 取り込み時に修正すべき既知不具合

world の現行実装には、仕様として採用する前に修正が必要な点があります。

- `World.addPlayer` と `setRunningStatus` が常に `false` を返す。
- `World.getPlayer(String id, String hash)` が `id` を使わず `hash` だけで検索する。
- `World.spendTime` はアクションが空でも `pop()` するため例外になる。
- アクションキューが周期ごとに最大60件ではなく、空キュー時の安全性がない。
- `distance` が差分の合計に対して平方根を取っており、ユークリッド距離になっていない。
- `CommandMaster.get` が自身を再帰呼び出ししている。
- `Player.process` が拡張for中にリスト要素を削除する。
- `Player.affect` が対象プレイヤーのステータスではなく、効果側のステータスを変更している。
- `World.raiseWorld` が再初期化時にグリッドを明示的にクリアしない。
- `World` の静的状態とワーカースレッドのライフサイクルが安全に分離されていない。
- `Grid` の静的マップがテスト間で共有され、リセットAPIがない。
- `Grid.affect` は対象グリッドがない場合にnull参照になる。
- `Element.Builder.fromJson` が `null` を返す。
- Javaコンパイラ設定は `<java.version>17</java.version>` としつつ、source/targetは8で不一致。

これらはC++への移植時に再現せず、受入条件に含めます。

## 取り込み単位

履歴と責務が衝突するため、既存の `seed-origin` のC++ソースへ直接上書きせず、次の単位で段階的に取り込みます。

1. `world-model`: `Element`、`WorldStatus`、コマンド定義、座標距離計算。
2. `world-grid`: グリッド生成、循環、6方向拡散、リセット可能な保管領域。
3. `world-effects`: `Buff` の寿命処理とプレイヤーへの安全な効果適用。
4. `world-actions`: 通常・管理アクション、制限判定、距離判定、アクションキュー。
5. `world-runtime`: 周期スレッド、開始・停止、ワールド状態集計。

最初の統合ではDB、MyBatis、JAX-RS、JSON通信を含めず、決定的なインメモリ処理として実装します。通信層はワールドコアの受入テスト後に別Issueとして扱います。

## 性能上の制約

`scode` の現行実装をそのまま移植してはいけません。20 × 20 × 20 の8,000グリッドを10ミリ秒ごとに全走査するため、理論上は毎秒800,000回の `Grid::circulate` 呼び出しになります。各呼び出しは最大6方向の連想コンテナ検索を行うため、セル数が増えると更新処理が急速に重くなります。

さらに、`Grid::getNextGrid` が存在しない近傍グリッドを遅延生成するため、初期範囲の境界で走査中にマップが変更されます。これはグリッド数の予期せぬ増加だけでなく、Javaの反復処理で `ConcurrentModificationException` を発生させる可能性があります。

初期C++実装では、次の条件を受入条件とします。

- 固定範囲を使う場合は、3次元配列または連続メモリ上のセル配列を優先する。
- 疎なワールドを使う場合は、元素量が変化するアクティブセルだけを更新対象にする。
- 周期処理中に近傍セルを生成・削除しない。境界外は存在しないセルとして扱う。
- 現在値を読み、次の値へ一括反映するダブルバッファ方式を基本とする。
- 環境循環の周期とアクション処理の周期を分離し、設定可能にする。
- 8,000セル全走査とアクティブセル更新の両方をベンチマークで比較する。

したがって、`world-grid` は機能移植ではなく、データ構造を含む再設計として扱います。性能計測で閾値を確認するまで `world-runtime` の常時スレッド化は行いません。

## 受入テストの骨子

- 既定サイズのグリッドが一度だけ生成され、リセット後に同じ状態へ戻る。
- 元素循環と6方向拡散が境界条件を含めて決定的に動作する。
- 既存プレイヤーをIDで追加・更新・削除できる。
- 管理アクションのログイン・ログアウトが安全に再実行できる。
- コマンド制限未達、距離超過、未知コマンドが副作用を起こさない。
- 成功したアクションだけが対象プレイヤーへ効果を付与する。
- 効果寿命が周期ごとに減少し、期限切れが安全に削除される。
- ワールドの開始・停止を複数回行ってもスレッドが増殖しない。
- 空のアクションキューを周期処理してもクラッシュしない。
